# World Structure

## 目标

当前项目已经补上 `area`，但如果后续继续扩地图，只停留在 `area + room`，仍然不够表达更大的世界层级。

因此后续世界结构建议固定为四层：

- `map`
- `zone`
- `area`
- `room`

这套结构的目的不是增加抽象数量，而是让世界观、配置结构和后续维护方式保持一致。

## 核心分层

### `map`

`map` 表示最大的世界板块。

例如：

- `神洲大陆`

它负责承载：

- 大世界名称
- 世界板块描述
- 大陆 / 海域 / 异界类型
- 其下有哪些 `zone`

### `zone`

`zone` 表示 `map` 下的大区域。

例如：

- `宁洲`
- `云州`
- `北荒`

它负责承载：

- 州域名
- 所属 `map`
- 大区域描述
- 推荐境界
- 区域性质
- 其下有哪些 `area`

### `area`

`area` 表示玩家会频繁进入和开发的具体地域。

例如：

- `青云渡新手村`
- `青云外门`
- `外门坊市`
- `山路试炼区`

它负责承载：

- 所属 `zone`
- 区域名
- 区域定位
- 推荐境界
- 区域描述
- 房间列表
- 基础设施
- 区域出口

### `room`

`room` 仍然是玩家实际站立和交互的最小地点。

例如：

- `渡口广场`
- `小药铺`
- `铁匠铺`
- `村口`

它负责承载：

- 具体描述
- 房间出口
- 当前 NPC
- 当前敌人
- 当前交互对象

## 推荐关系

推荐固定为：

- `map` 包含多个 `zone`
- `zone` 包含多个 `area`
- `area` 包含多个 `room`

推荐示例：

- `神洲大陆`
  - `宁洲`
    - `青云渡新手村`
      - `青云渡`
      - `问道石阶`
      - `古松林`
    - `青云外门`
      - `溪谷栈道`
      - `青云外门前坪`
      - `听泉药圃`

## 推荐配置文件

后续内容层建议固定这几份：

- `world/data/maps.json`
- `world/data/zones.json`
- `world/data/areas.json`
- `world/data/rooms.json`
- `world/data/area_exits.json`

并维持这些引用关系：

- `zones.json` 通过 `map_id` 归属 `map`
- `areas.json` 通过 `zone_id` 归属 `zone`
- `rooms.json` 通过 `area_id` 归属 `area`

## 为什么值得现在做

如果继续只做到 `area`：

- 世界观设定会停留在“一个个村镇和山门”
- 后续很难自然表达“这是同一州域里的不同区域”
- 大地图、州界、州内势力、跨州迁移后面会临时补结构

如果现在补 `map / zone`：

- 后续新增地域有统一挂点
- 区域导航会更自然
- 世界设定文档和代码结构能对齐
- 商店、门派、世界事件、跨州迁移更容易接入

## 当前第一版落点

当前推荐先固定第一版世界结构：

- `map`
  - `神洲大陆`
- `zone`
  - `宁洲`
- `area`
  - `青云渡新手村`
  - `青云外门`

这一步先只做：

- 文档定稿
- 配置文件落地
- `area` 挂 `zone_id`

暂时不急着同时做：

- 跨州迁移
- 大地图 UI
- 世界事件
- 州级势力系统

## 设计原则

1. `map / zone` 主要用于组织世界结构，不替代 `area / room` 的实际游玩分层。
2. 玩家日常游玩仍然主要发生在 `room` 与 `area`。
3. `zone` 适合承载州域设定、区域推荐和中层导航。
4. `map` 适合承载大世界设定与后续跨大陆扩展。
5. 后续新增地域时，优先先确定它属于哪个 `map`、哪个 `zone`，再落 `area` 和 `room`。
