# Area Design

## 目标

当前项目已经有若干 `room`，但后续继续扩世界时，只靠房间会逐渐失去“地域层级”。

因此后续地图结构增加一层更高的内容单位：

- `area`: 地域 / 区域
- `room`: 区域内部的具体地点

这一步的目标不是替代 `room`，而是让地图开始具备：

- 区域归属
- 区域级设施规划
- 区域到区域的迁移关系
- 商店、村舍、药铺、铁匠铺等功能设施的稳定挂载点

## 核心分层

### `area`

`area` 表示一个完整地域，例如：

- `青云渡新手村`
- `青云外门`
- `山门外道`
- `坊市`

它负责承载区域级信息：

- 区域 `id`
- 区域名
- 区域定位
- 推荐境界
- 进入条件
- 区域描述
- 包含哪些 `room`
- 有哪些基础设施
- 有哪些区域出口

### `room`

`room` 仍然保留为玩家实际进入、移动和交互的最小地点。

例如一个 `area` 内可以包含：

- 渡口广场
- 小药铺
- 铁匠铺
- 村舍
- 村口
- 仓房

`room` 负责：

- 地点描述
- 房间内出口
- 当前 NPC
- 当前敌人
- 当前对象
- 房间级交互

### `area_exit`

`area_exit` 用于表示“区域到区域”的迁移点，而不是单纯的房间出口。

例如：

- `青云渡新手村 -> 青云外门`
- `青云外门 -> 山路试炼区`
- `青云外门 -> 外门坊市`

它负责：

- 起始区域
- 目标区域
- 实际触发房间 / 对象
- 解锁条件
- 迁移提示文本

## 推荐的数据结构

后续建议新增这些配置文件：

- `world/data/areas.json`
  - 定义区域元数据
- `world/data/area_exits.json`
  - 定义区域间迁移关系
- `world/data/shops.json`
  - 定义商店、商人、商品和价格

同时扩展现有配置：

- `rooms.json`
  - 给每个房间增加 `area_id`
- `npcs.json`
  - 后续可给商店 NPC 增加 `shop_id`
- `objects.json`
  - 后续可给区域入口对象增加 `area_exit_id`

## 第一个标准区域

建议先把当前“新手区”正式整理成第一个 `area`：

- 区域名：`青云渡新手村`
- 区域 ID：`area_qingyundu_village`

建议归入的房间：

- `青云渡`
- `问道石阶`
- `古松林`

建议补充的设施房间：

- `小药铺`
- `铁匠铺`
- `村舍`
- `村口杂货摊`

建议挂载的功能：

- 新手引导 NPC
- 基础补给商店
- 基础回复资源
- 基础采集点
- 区域出口

## 商店系统定位

商店建议作为 `area` 的标准基础设施，而不是零散房间功能。

第一阶段只做最小商店：

- 基础货币字段，例如 `铜钱`
- 基础购买命令
- 少量新手补给商品

推荐最小商品：

- `松纹草`
- `渡口药包`
- 基础材料

第一阶段先支持：

- `商店`
- `购买 <物品>`

暂不急着做：

- 售出回收
- 动态价格
- 税率
- 库存刷新

## 为什么现在就要加 area

如果继续只按 `room` 堆内容，短期能跑，但后面会逐渐遇到这些问题：

- 很难表达“这是一个完整新手村”
- 很难做区域级解锁
- 很难做区域推荐和迁移
- 商店、门派入口、区域任务会缺少统一挂点
- 地图扩张后维护成本会快速升高

所以 `area` 是当前非常值得补的一层基础结构。

## 设计原则

1. `area` 是组织层，不替代 `room`。
2. 玩家移动仍然主要发生在 `room` 之间。
3. 区域迁移优先走 `area_exit`，而不是散落在任意房间出口里。
4. 商店、药铺、铁匠铺、村舍等功能设施优先作为 `area` 下的标准组成部分。
5. 新区域上线时，优先先定义 `area`，再补 `room`、NPC、对象和任务。

## 近期实施顺序

建议后续按这个顺序推进：

1. 新增 `areas.json`
2. 给 `rooms.json` 增加 `area_id`
3. 新增 `area_exits.json`
4. 把 `青云渡新手村` 建成第一个标准区域
5. 新增最小商店系统

## 当前结论

这套设计非常适合当前项目阶段。

它不会推翻现有 `room` 体系，只是在其上增加一层更利于维护和扩图的结构。
后续只要按照这个模式继续扩展，新的村镇、外门、山林、坊市都会更容易维护。
